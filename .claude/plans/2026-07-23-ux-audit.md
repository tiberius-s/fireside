## Executive summary

The two prior audits — and this session's own `scripts/verify.sh` run —
already established that the presenter, the CLI verbs, and the editor's
scripted 10-minute flagship smoke walk are solid. This pass's job was to
find what a scripted smoke test wouldn't: unscripted, exploratory use of
`fireside edit`. It surfaced two real findings, both concentrated in the
editor's two least-exercised corners — its opening rules and its
container-block handling — plus one small wording papercut carried over
from the presenter/notes work. Nothing rises to the "first five minutes"
severity of the 2026-07-19 audit's P0s; both real findings are P1s because
they're about an author quietly ending up in the wrong place with real
work, not about the app refusing to start.

## Wave 1 — P1: editor correctness

**P1-1: `fireside edit <path>` on a mistyped path silently creates and
writes a brand-new starter deck — no error, no visible confirmation.**

Observed: `fireside edit my-real-talk.fireside.json` (an existing 8 KB
deck) works fine. `fireside edit my-real-tlak.fireside.json` (two letters
transposed) does **not** error — it silently writes a new 2.6 KB starter
deck at the typo'd path and opens *that*, titled "my real tlak" from the
filename. Verified reproducibly, including at a 30ms capture right after
launch: `edit.rs::create_deck` does print `No deck at {path} yet —
creating a starter deck to edit.` (`crates/fireside-cli/src/edit.rs:178`),
but it's a `println!` before `try_init()` — the very next frame is the
alternate-screen TUI, which overwrites it before a real terminal ever
renders it. This is the same transient-flash pattern `fireside new
<name>` already uses (`new.rs:50`, `println!("Created {}.", ...)` right
before `present()` takes over the screen) — not a new pattern, and not a
bug in isolation.

What makes it a real trap specific to `edit` rather than a shrug: `present`
and `validate` refuse a missing path outright (`No deck named X —
"fireside new X" creates one.`, P0-2/W1-4 from the prior audits) — a typo
there is loud and immediately corrected. `edit`'s create-if-missing
(contract rule 4, `contracts/cli-edit-command.md`) is deliberately
different — an actual flow, not a refusal — which is right for a
genuinely new file, but has no way to distinguish "you meant to create
this" from "you meant an existing file and fat-fingered the name." Because
an editor session is long-lived (unlike `present`, where showing the wrong
deck for two seconds is obvious and cheap to back out of), an author can
type real content into the wrong file for a while, hit `[ Save ]`, and now
has unsaved effort sitting in a file they didn't mean to touch, while the
real file remains untouched and looks like nothing happened. Two stray
files (`newtalk.fireside.json`, `my-real-tlak.fireside.json`) were left
behind by nothing more than routine typing during this audit.

Who: authors, especially returning to edit a deck days later from memory
or shell history.

Fix (shape, not final): before creating, `edit` already knows the file
doesn't exist — it could look for a near-miss among `*.fireside.json`
files in the same directory (cheap: `std::fs::read_dir` + a short
edit-distance or prefix check) and, if found, ask
`No file at X. Did you mean Y? [ Open Y ] [ Create X ]` instead of
creating unconditionally. Simpler alternative that fixes the "invisible
confirmation" half even without the near-miss check: hold the create
message on screen — either don't enter the TUI until a keypress, or paint
it as the editor's first flash message (`EditorApp::set_flash`, already
exists) instead of a pre-TUI `println!`, so "this is a brand-new file" is
something the author actually sees inside the tool they're now using.
Either fix is CLI/editor-only, no protocol or crate-boundary change.

**P1-2: A container block's children can never be selected, edited,
reordered, or deleted in the editor — only the container as a whole.**

Observed: on the bundled demo deck's "Welcome" slide (a single top-level
`container` holding a text-art title, a tagline, a divider, and closing
text), `Tab` selects the container as one unit and stays there — it never
descends into or cycles through the container's children, because
`select_adjacent_block` iterates `node.content.len()` (top-level blocks
only: 1, for this slide). Clicking directly on any child's own rendered
text on the canvas (verified: clicking exactly on the "Fireside" text-art
line) also selects the whole container, never the child, because
`hit::block_extents`' own doc comment says so explicitly: "Nested
(`Container`) children are out of scope for the canvas... container
children are reached through the container form's breadcrumb, T033"
(`crates/fireside-tui/src/editor/hit.rs:470`). Opening the container's own
`[ ✎ Edit ]` form does show a plain-language, one-line-per-child summary
list (`ChildSummary`, "• text art — Fireside", "• text — ...", etc.) — but
it is display-only. `Down`/`Enter` inside that form do nothing; clicking
directly on a bullet row (verified at its exact terminal coordinates) also
does nothing. That form's own doc comment says the opposite of `hit.rs`'s:
"drilling in to *edit* a child is left to the canvas's own block selection
once US2 extends hit-testing past top-level blocks"
(`crates/fireside-tui/src/editor/forms.rs:225`). Each side's comment
names the *other* side as where this is implemented; neither actually
implements it. The only thing a container's form can do today is cycle
its `Layout` (Stack/Columns/Centered) and commit that immediately — the
children inside stay exactly as authored, forever, once a deck has one.

Impact, concretely: 4 of the bundled demo deck's 8 slides (`welcome`,
`layout`, `extras`, `finale`) are built from containers; every word of
text inside every one of those slides — including the "Layout" slide,
whose whole purpose is to *show off* container layouts — is unreachable
from the block editor. This isn't an edge case; a container is one of the
editor's own 8 documented block kinds (`guides/editing.md`'s own child
form list mentions it), and "columns/box/stack" layout is a natural choice
for any title slide or side-by-side comparison — exactly the kind of
slide a no-JSON author would reach for.

Who: any author who builds (or opens a deck that already has) a
columns/box/stack slide — including, ironically, `fireside edit`'s own
best demo of itself.

Fix (shape, not final): extend `BlockPath` selection/hit-testing to
descend into a container's children — `Tab`/`Shift+Tab` already carries a
`Vec<usize>` path (`select_adjacent_block` just never grows it past one
element), and `hit::block_extents` would need a recursive variant for a
selected container's own rendered sub-range. This is squarely a `US2`-
scope completion (per the forms.rs comment's own framing), not a new
story — `contracts/hit-testing.md`'s `BlockPath`/`Target::Block` shapes
already generalize to nested paths; the gap is that nothing walks past
depth 1. Needs its own tasks (hit-testing, canvas selection glow inside a
container, the child-edit-form wiring, tests at all three constitution-VII
layers) — small-to-medium, not a redesign.

## Wave 2 — P2: papercuts

**P2-1: The non-tty refusal always says "to present," even from `edit` or
`notes`.**
Observed: `echo | fireside edit x.fireside.json` (and `notes`) print
`fireside needs an interactive terminal to present — run it directly in
your terminal, not through a pipe.` — the exact same
`TuiError::NotATty` message `present` uses
(`crates/fireside-tui/src/error.rs:20`), worded for the verb it was
written for first. Not misleading enough to block anyone, but noticeably
verb-mismatched once you notice it. Fix: drop the verb-specific half —
`fireside needs an interactive terminal — run it directly, not through a
pipe.` reads correctly for all three callers with no parameterization
needed.

## Docs

`README.md`, and `quickstart.md`, read correctly and match current
behavior — spot-checked against the actual `?` help overlay text and the
toolbar/hint-line chip labels, no drift found. One thing worth noting for
whoever picks up P1-2: `guides/editing.md`'s "Selecting and editing"
section already describes per-kind forms for all eight block kinds

columns/box/stack containers") — accurate as far as it goes (the layout

themselves aren't reachable. Once P1-2 has a fix or an accepted scope
decision, this page needs a matching line either way (documenting the
real capability, or the real limitation).

## Suggested order

P1-1 and P1-2 are independent and both CLI/TUI-only (no protocol, no
crate-boundary change) — either can go first. P1-1 is the smaller change
(a message-visibility fix, optionally a near-miss suggestion). P1-2 is the
more consequential one — it's a real gap in the "no JSON required" promise
spec 013 exists to deliver, on a block kind the editor's own demo deck
leans on — and should get a proper spec-kit-scoped look (even if small)
rather than a quick patch, since it touches hit-testing and selection
state shared across the whole editor. P2-1 rides along with either.
